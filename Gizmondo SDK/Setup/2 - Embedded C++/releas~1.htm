System Requirements

Windows CE .NET has the following system requirements:

Windows CE .NET System

Requirements

To use Microsoft Windows CE .NET, you need:

  • 266-MHz Pentium II processor or higher; 500-MHz processor recommended.
  • Microsoft Windows 2000 Professional with Service Pack 2 or Windows XP Professional.
  • 128 MB of RAM; 192 MB of RAM required to run the Emulator.
  • 1.7 GB of available hard-disk space for a typical single microprocessor installation; 14 GB for installation of the entire product, including the Platform Builder toolset.
  • CD-ROM or DVD-ROM drive.
  • VGA or higher resolution monitor; Super VGA recommended.
  • Microsoft Mouse or compatible pointing device.
  • Bidirectional parallel port to download an OS image when not using an Ethernet connection.
  • Serial port or Ethernet network card for debugging support; a local area network (LAN) hub is recommended.

Windows CE .NET Installation

Windows CE .NET is distributed on multiple CDs or a single DVD. The setup application, Setup.exe, copies the files from the CDs or DVD to the development workstation.

It is recommended that you format your hard drive and reinstall the operating system prior to installing Windows CE .NET.

To install Windows CE .NET

  1. Log on to the development workstation with Windows administrator privileges.
  2. Run Setup.exe from the root directory on the Windows CE .NET CD number 1 or on the DVD.
  3. When the Welcome dialog box appears, choose Install, and then follow the on-screen instructions.
    Note   To reduce the installation size of Platform Builder, you can select only the microprocessors you need in the Custom Setup dialog box.

Installing Windows CE .NET on a computer containing Platform Builder 3.0, Talisker Beta 1, or Talisker Beta 2 may result in invalid registry keys, file version conflicts, or other problems. To avoid these problems, uninstall Platform Builder 3.0, Talisker Beta 1, or Talisker Beta 2 before installing Windows CE .NET.

Do not upgrade from Talisker Beta 2 to Windows CE .NET. It is recommended that you first uninstall Talisker Beta 2, and then install Windows CE .NET.

During the installation from a CD or DVD, the progress bar is not always accurate, especially for larger installations with more than one CPU.

Installation times can vary from the following:

  • 20-30 minutes for a Tools Only install
  • 30-40 minutes for a default installation (Tools, Test Kit, CEPC, and Emulator)
  • 2 hours for a full install
    Note   You must have administrative privileges to install and run Windows CE .NET.

Application Issues

ActiveSync is blocked when IrDA and OBEX are present simultaneously

ActiveSync is blocked when IrDA and OBEX are present simultaneously

ActiveSync on the device (Repllog.exe) needs the serial port (COM1) to work with the default connectoid. This default connectoid called Desktop @ 19200 uses COM1 to connect to the desktop. The IrDA DLL (Irdastk.dll) uses COM1 by default unless setup otherwise and the OBEX server by default will open a socket to listen on this IrDA port. Thus, COM1 is denied to ActiveSync causing it to generate a Port Not Available error. This will happen on the PDA configuration and in any configuration where IrDA and OBEX are present together and are using the default configuration.

Workarounds are to set up OBEX to use other transport mechanisms or reconfigure IrDA. The other option is to reconfigure ActiveSync to avoid using the default connection. Options are to configure ActiveSync to use other COM ports if available. This can be done by using the making new connectoids: Use the Make new Connection application found under the Network and Dial up Connection Control Panel application. Use the Direct Connection option to choose other COM ports. The other option is to use ActiveSync over USB instead of over default serial port. This is accomplished in a similar manner but in the Direct Connection option choose the USB serial port. Note that this needs the availability of a USB function port and a USB serial function driver on the device.

Note   ActiveSync over an Ethernet connection will not work for the initial connection. An Ethernet connection can be used after synchronizing with USB once to set up the initial partnership on the desktop side.

If you use the parallel download service to download an OS image and you use the Ethernet kernel transport, you must specify the correct I/O base address for the Ethernet adapter as an argument for the Loadcepc.exe executable on the target device. The Ethernet kernel transport does not autodetect the I/O base address. To obtain the correct I/O base address, download an OS image once to the target device using the Ethernet download service and not the I/O base address. Use the parallel download service for subsequent downloads.

Adding RNDIS to your platform

The following steps show how to enable the RNDIS host and function drivers:

  1. For the RNDIS function and RNDIS host driver, clear the BSP_NORNDIS and BSP_NOUSB environment variables.
  2. For the RNDIS function driver, set the BSP_RNDISMINI_NET2890 and SYSGEN_ETHERNET environment variables.
  3. For the RNDIS host driver, set the SYSGEN_ETH_USB_HOST and SYSGEN_USB environment variables.

Debugger sometimes has problems stepping large case statements

If the data in a jump table generated by the compiler for switch statements just happens to disassemble into a branch, the debugger might have problems stepping properly. Larger switch statements have more opportunities for generating this data and therefore provide more opportunity for the debugger to do the wrong thing.

As a workaround, use breakpoints in lieu of stepping large switch statements.

Debugger stops responding when unloading symbols while running

If you have variables in the Watch window and you attempt to unload symbol files in the Modules and Symbols window while the debugger is running, the Platform Builder debugger will stop responding.

Note   This problem is not consistent across platforms.

Resetting the target just after connecting the debugger generates a connection error

If you attempt to reset your target device immediately after connecting the debugger, you will likely interrupt the target while it is performing a load notification. Then, the debugger shows a communication error dialog box after about 15 seconds. From the dialog box, you have to choose Abort, close the debugger, restart Platform Builder, and then manually reset the target device. To avoid this, connect to the debugger after the target device is reset or wait for all modules to load before resetting the target.

Limitation of instantiating BP on new module loads

The following lists the current limitations of instantiating BP on new module loads:

  • You can only instantiate a source BP on a new instance of an existing module if the currently existing BP was instantiated on one module at a time only. If the BP was already instantiated on multiple BPs simultaneously, at once, or after a secondary module load, the BP will not be splittable anymore.
  • When a line contains BPs of multiple instances of the same module, and not all BPs are at the same instantiation state (deferred, instantiated, and disable), they will be shown in only one state, which is not defined.

Stepping out of C++ catch blocks and __finally blocks on x86

On x86 platforms, when you are presented with the opportunity to single step or range step out of a catch block, the step will take you out of your current application and into a C++ exception handler routine in Nk.exe. The expected behavior is to step out of the catch block and to resume debugging your application at the next instruction after the try{...} catch(){...} structure. However, attempting to step through the handler code will not return you to the excepted location in the application being debugged because the handler routine mangles the stack. This behavior also manifests when stepping out of a __finally block while unwinding exceptions for the same reason.

You can work around these limitations by stepping into the block, setting breakpoints at an appropriate location in the program that will execute later, and then using a go to run to the breakpoint.

Stepping in eXDI with Auto-Refresh on module window will request a full reload module every time

Using a Step command while in eXDI Hardware-Assisted Debugging causes the debugger to request a full module reload if the Auto-Refresh on Step mode in the Modules window is enabled. To avoid this, you must disable the Auto-Refresh on Step mode in the Modules window.

PPFS holds files open after the Debug and Stop Debugging commands

PPFS, which is responsible for making files from the development workstation's flat release directory available on the Windows CE device, holds executable files open as long as they are running. When an executable file is opened in this fashion, it cannot be overwritten or deleted simply by compiling a new version on the development workstation. The file is held open until the executable has run its course and is terminated or the device is reset. In particular, the Debug and Stop Debugging commands are not sufficient to close handles to the file in the flat release directory because the handles will remain open until the file completes its execution. This results in an error stating that the executable cannot be opened if the project has been recompiled after a Debug or Stop Debugging command has been entered.

Deferred BP in a file that does not exist anymore will generate infinite request to resolve ambiguity

If a mapping is generated for a source file that later is deleted or moved from the mapped location and a new mapping for this source is performed, an ambiguous source dialog box is generated and cause some confusion.

The key is to recognize that the listed file is no longer available at that path and the proper response is to dismiss the dialog box.

SRMP registry settings

MSMQ installation

Make sure that your Windows CE device has a unique name on the network before installing MSMQ.

MSMQ message routing

Message Queuing provides a set of routing registries that enable messages to be routed in a variety of ways in response to network conditions.

Note   The router registries must be set before the MSMQ service starts.

You can use the HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\SimpleClient registry key to set the message route. The following subkeys specify the message routes:

  • RouteTo
  • RouteFrom
  • RouteLocal

Use the RouteTo and RouteFrom pair when the source and destination MSMQ hosts are not exposed directly to each other in the network. You can specify a router that connects to both source and destination hosts to send messages across.

For example, to allow abc123.com to send messages to xyz.com and they are not exposed directly to each other in the network, you can set the RouteTo subkey of abc123.com to use def456.com as the router. The following registry setting allows the application to deliver all messages whose original destination is xyz.com to the new destination def456.com.

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\SimpleClient\RouteTo]
    "DIRECT=OS:xyz.com\*"="DIRECT=OS:def456.com\routingqueue"

The new host, def456.com, creates routingqueue as needed, so you do not have to create this queue. The original host value, as in DIRECT=OS:xyz.com\*, allows you to specify the host name (xyz.com) but not the queue name. Only the host name part, xyz.com, is used for the routing. Therefore, all queues going to the xyz.com will be routed to def456.com.

The delivery of messages to xyz.com totally depends on the routing ability of the MSMQ message router (def345.com) once those messages are sent to the router. MSMQ in Windows CE .NET has a routing capability, so messages can be routed to their original destination if def456.com is exposed to abc123.com. Windows CE 3.0 and earlier and the desktop version of MSMQ do not support this message routing. If the router does not have a routing capability, the messages will simply be discarded.

Note   Messages being routed by RouteTo and RouteFrom still maintain their original destination queue name.

This registry setting is useful when the MSMQ sender cannot directly reach the destination host device, such as a device behind a firewall or when connected to another network protocol. In this case, you can set the firewall or the edge device as the router.

The RouteFrom subkey is used to handle special cases. Sometimes you may want to forward messages only from a known MSMQ host for security purposes. You set the RouteFrom subkey in your edge device. This way you can specify the source devices that you allow the edge device to route messages for.

For example, to allow ghi.com to route messages from abc123.com to the final destination xyz.com, you set the RouteFrom subkey of ghi.com to following registry setting.

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\SimpleClient\RouteFrom]
    "f10436ce-8e8c-a581-1a84-e10f0ce265bf"="DIRECT=OS:xyz.com\routingqueue"

Note that the GUID of the sender (abc123.com) is f10436ce-8e8c-a581-1a84-e10f0ce265bf.

SRMP routing uses the RouteLocal subkey for message routing. This is not related either the RouteTo or RouteFrom subkeys.

For example, if abc123.com sends a message to the DIRECT=HTTP://def456.com/msmq/private$/Q1, and you want forward the message to another destination without notifying the sender, you can set the abc123.com registry to the following value

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\SimpleClient\RouteLocal]
    "URI1"="DIRECT=HTTP://def456.com/msmq/private$/Q1"

Then you can set the def456.com registry to the following value.

[HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\SimpleClient\RouteLocal]
    "URI1"="DIRECT=HTTP://xyz.com/msmq/private$/account"

Whenever abc123.com sends an SRMP message to "HTTP://def456.com/msmq/private$/Q1", this is sent to the def456.com with a tag "URI1". When def456.com receives this SRMP message, def456.com checks its local registry for matching "URI1". If the def456.com finds a registry setting, "DIRECT=HTTP://xyz.com/msmq/private$/account" in this case, it forwards that message to the specified queue. The RouteLocal subkey applies only when sending messages through SRMP. The routing destination in the destination-side registry can be any valid queue name.

Note   Using RouteLocal actually changes the final destination queue name in the message.

The following sample code shows how to set the RouteTo subkey.

WCHAR* pSource = L"DIRECT=OS:cemsmq-wxp\\*";
WCHAR* pDestination = L"DIRECT=OS:CEPCKS32976X\\routingqueue";
WCHAR* pRegKey = L"SOFTWARE\\Microsoft\\MSMQ\\SimpleClient\\RouteTo"
HKEY hKey = NULL;
DWORD dwDisp = 0;
if(ERROR_SUCCESS == RegCreateKeyEx(
                         HKEY_LOCAL_MACHINE,
                         pRegKey,
                         0,
                         NULL,
                         REG_OPTION_NON_VOLATILE,
                         KEY_WRITE,
                         NULL,
                         &hKey,
                         &dwDisp))
{
   if(dwDisp == REG_OPENED_EXISTING_KEY)
      printf("MSMQ: registry key exist (overwriting)");

   ULONG hr = RegSetValueEx(
                 hKey,
                 pSource,
                 0,
                 REG_SZ,
                 (BYTE *)pDestination,
                 sizeof(WCHAR)*(wcslen(pDestination)+1));
   RegCloseKey(hKey);
   if(hr!=ERROR_SUCCESS)
      printf("error 0x%0X", hr);
}

Platform Manager KITL Application Wizard is dependent on system variables

You must set the PLATMANROOT and OSROOT environment variables before building a COM wrapper for a kernel transport with the Custom KITL Application Wizard or before building a COM wrapper for a custom download service with the Custom Download Application Wizard. Set the PLATMANROOT variable equal to the path of the root-level Platman directory for Platform Manager. Set the OSROOT variable equal to the path of the root-level Wince400 directory for the Windows CE OS.

Desktop firewall interferes with Remote Tools over TCP/IP

If the development workstation has a firewall, a remote tool is unable to connect to a target device using the TCP/IP transport. The firewall blocks the information required to establish the connection.

To allow the remote tool to connect to a target device using the TCP/IP transport, disable the firewall. Alternatively, in configuring the TCP/IP transport, designate a fixed port through which to communicate from the target device to the development workstation. After you select a fixed port, configure the firewall to allow information to be received on that port.

Remote Tools do not work with polled mode KITL

On a platform that uses polled mode KITL, such as a platform that uses the Assabet board support package (BSP), you cannot establish a connection from a remote tool to a target device using the KITL transport for Windows CE .NET. You must instead use the TCP/IP transport and connect to the product Ethernet card on the target device to establish a connection.

Platform Manager samples are not compatible with Visual Basic .NET

The Visual Basic sample code provided for use with the Platform Manager was written using Visual Studio 6.0. It will not work in Visual Studio .NET.

Connecting Remote Tools over the serial transport

If you use the serial kernel transport to connect to a target device, take the following steps to connect any remote tool except the Remote System Information tool to the target device.

  1. After establishing a connection to the target device, open a remote tool.
  2. From the Connection menu in the window for the remote tool, choose Configure Windows CE Platform Manager.
  3. Select your device, and then choose Properties.
  4. In the Transport box, choose KITL Transport for Windows CE, and then choose Configure.
  5. Clear the Check connection status check box, and then choose OK.
  6. Choose OK, and then choose OK again.

After completing these steps, you can connect any number of remote tools to the target device. You do not need to repeat these steps for each remote tool that you connect to the target device.

Do not use the Remote System Information tool with the serial kernel transport. The Remote System Information tool attempts to use the serial driver built into the OS image and disrupts serial communications between the development workstation and the target device.

Emulator for Windows CE .NET

When you are using the Emulator your entire platform is running in software. This means that in almost all cases performance on the Emulator will be slower than on actual hardware. For many operating system tasks a degradation of performance will not affect the ultimate success of those tasks, even though they may appear halted at times. However, operating system tasks with a time-critical aspect to them may not be successful when running on the Emulator. For instance, real-time applications and development are not supported in the Emulator, and timing-sensitive applications, like multimedia, may show degraded or erratic performance.

Strictly speaking, the Emulator only supports x86 platforms. In actual practice most OS APIs will run identically on other processors. Also, developing on the Emulator gives no real advantage to deploying to x86 hardware in comparison with other processors.

The recommended system requirements for running one instance of the Emulator configured to use 64 MB of memory are as follows:

  • 1-GHz CPU
  • 384 MB of system memory
  • 500 MB of free space on hard drive

Each additional instance of the Emulator requires an additional 32 MB of system memory plus an amount of system memory equal to the amount of memory that you configure the Emulator to use.

Applications and services development

  • No support for the Simple Network Management Protocol (SNMP).
  • The Emulator can be used with ActiveSync 3.5, but earlier versions of ActiveSync will not work with the Emulator. You can use two serial ports connected together with a null modem cable to provide the communications path between Platform Builder and the Emulator. The following steps describe how to set this up.

Configuring the Emulator for ActiveSync

  1. Choose Emulator from the Download list on the Services tab of the Configure Remote Connection dialog box.
  2. From the Download list, choose Configure.
  3. In the Configure Emulator Download Service dialog box, set the serial port that the Emulator will use. Choose COM1 or COM2, whichever you prefer, from the Serial Port 2 list if KITL is built into your OS image.

    - or -

    If KITL is not built into your OS image, choose COM1 or COM2, whichever you prefer, from the Serial Port 1 list.

  4. Choose None from the serial port list that you did not assign a COM port to in the previous step.
  5. Choose OK to close the Configure Emulator Download Service dialog box.
  6. Choose OK to close the Configure Remote Connection dialog box.
  7. On the desktop, configure ActiveSync 3.5 connection settings to connect to whichever serial port, COM1 or COM2, was not assigned to the Emulator.
  8. On your desktop computer, connect the serial ports COM1 and COM2 together with the null modem cable.
  9. Once your platform is downloaded to the Emulator, run Repllog.exe on the target device in the Emulator. It should connect. Occasionally the connection will fail. If this happens, wait for Repllog.exe to time out and then retry it.

Communications and networking

The following features work with limitations in the Emulator:

  • The Emulator does not allow incoming connections. Other applications will not be able to connect to the CE device running inside the Emulator. The following components only work for outbound connections:
    • DCOM
    • MSMQ
    • SOAP
    • OBEX
  • The Emulator cannot act as a network sever. It has its own private IP address and is therefore unreachable from outside applications. The servers will work locally within the Emulator. For example, calls to http://localhost will be successful.
    • Web Server
    • RAS Server
    • Telnet Server
    • FTP Server
  • Internet 5.5 exhibits poor performance for the following scenarios:
    • Loading graphics or content-rich Web pages
    • Scrolling a Web page
  • Dial-Up Networking only works with an external modem. Dial-Up Networking does not work in the Emulator using a software modem, IrDA, or Bluetooth.
  • You can set up a PPTP connection over a PPP connection, such as Dial-Up Networking, using a modem or with a direct connection using a null modem cable.
  • In the Emulator, it is not possible to set up a virtual private network (VPN) connection using PPTP over a local area network. This includes both Ethernet and 802.11. Attempting to create a VPN connection will fail with a message saying that the server dropped the connection.
  • The Emulator does not provide network access when the host computer is connected to the network through a VPN connection.

The following features do not work in the Emulator:

  • IrDa
  • Bluetooth
  • 802.11
  • Modems
  • No voice over IP (VoIP) in Windows Messenger because the Emulator only supports half-duplex audio.
  • Real-Time Communication (RTC) does not support connection over the Universal Datagram Protocol (UDP).

Hardware support

The following devices are supported in the Emulator:

  • Serial ports. You can connect modems through serial ports COM1 and COM2.
  • Parallel ports. You can connect a printer to the parallel port and print from WordPad.
  • The Emulator provides network support by emulating the DEC 2114x driver in software. No other Ethernet drivers are supported.
  • The Emulator provides display support by emulating the VGAFLAT driver. No other display drivers are supported.
  • The Emulator provides audio support by emulating the Soundblaster 16 wavedev driver. This driver only supports half-duplex audio. No other audio drivers are supported.

The following devices are not supported in the Emulator:

  • USB devices
  • PC Card devices
  • Storage devices, including CF cards and hard disks
  • File system drivers, including FAT, CD, and DVD
  • IEEE 1394 devices

Multimedia technologies

The Emulator must compete with the host machine for the CPU and other hardware resources. This can have a severe affect on multimedia tasks, which are generally performance-sensitive and can at times be CPU intensive. Moreover, multimedia performance as well as selected features can depend on having appropriate hardware acceleration available on the platform.

The Emulator does not support hardware acceleration for multimedia. Although some multimedia features have their own software emulation capabilities built into them, even these features will experience performance or driver limitations on the Emulator. Some important multimedia features are specifically hardware-related and are never supported in software even in real Windows CE devices.

The following features work with limitations in the Emulator:

  • Windows Media playback performance is highly dependant on the processing power of the host system.
  • DirectShow works within the limitations imposed by the audio and display drivers as well as the processing power of the host system.
  • DirectSound works within the limitations imposed by the audio driver. The Emulator does not support DirectSound capture.
  • The audio driver in the Emulator is limited to 22.05-kHz native output and only runs in half-duplex mode. Despite this limitation, with the unified audio model (UAM), it is possible to play back audio data that has been recorded at other sample rates, such as 44.1 kHz. Down-sampling does have an affect on audio quality, making it slightly muddier or duller.
  • Waveform Audio capture quality is poor.
  • You can encounter some minor audio playback issues when the Emulator must interact and compete with Windows XP for access to audio hardware on the host machine.
  • A number of use scenarios related to moving or changing display elements on the screen can cause minor ticks or disruptions during audio playback. Intense graphics changes, such as with Direct3D, may result in poor audio quality or even block audio output entirely.
  • The Emulator only has a DirectDraw-enabled FLAT driver. There is no emulation support for hardware features such as hardware blits, flips, overlays, or video ports.
  • Direct3D inherits the display driver limitations imposed on DirectDraw. Additionally, because there is no support for hardware acceleration in the Emulator, all rendering must be done through the reference rasterizer so you can expect slow performance.

The following features do not work in the Emulator:

  • The DVD-Video API will not work in the Emulator. Although it is possible to implement a software-based decoding solution, the DVD-Video API is delivered with the assumption that all DVD-Video decoding will be done in dedicated decoding and rendering hardware. The Emulator does not have any support for DVD-Video decoding hardware.
  • Digital Rights Management (DRM) is not supported in the Emulator due to the inherent complexities associated with security and the necessary OEM-specific DRM libraries. Trying to play a DRM clip in the Emulator will open a dialog box indicating that DRM is not included in your platform and will cause playback to gracefully fail.

Shell and user interface

The following feature works with limitations in the Emulator:

  • Scrolling and display is slow, affecting all applications.

The following features do not work in the Emulator:

  • Screen rotation.
  • The Emulator cannot emulate multiple screen support for a Windows CE device.

Platform Builder

Some features of Platform Builder can have an affect on how an otherwise supported platform runs in the Emulator.

The Emulator can run OS images from two places: from Platform Builder or in a development environment as part of a rolled SDK. The Emulator will work with Microsoft eMbedded Visual C++ 4.0. It will also work with Microsoft eMbedded Visual Tools 3.0, but only if the PocketPC 2002 SDK has also been installed.

If you are using the Emulator in Platform Builder, you can include KITL in your image. If you are creating an image as part of an SDK, KITL should not be included in your image. If KITL is included, the Emulator will block the development tools as it launches and then waits for a KITL connection that can never be made.

If you are running in the Emulator and try to connect using a Remote Desktop client to the same desktop system the Emulator is running on, you will freeze the desktop screen after you have supplied your credentials.

The following board support package (BSP) issues affect the Emulator:

  • When you add the Emulator BSP to a project, the network driver changes from DEC 2114x to Realtek RTK8139. However, once the project is built the DEC driver will appear in Control Panel of the device running on the Emulator.
  • When you use the New Platform Wizard to create a Web Pad platform, the USB Human Interface Device (HID) driver is added by default even though USB hardware is not supported.

Platform Builder upgrades cause the Emulator Download dialog box to display invalid data for existing devices

If you upgrade to Platform Builder 4.0 from a beta version of Platform Builder and a named connection in the beta version uses the Emulator download service, the information for the Emulator download service for that named connection becomes invalid after the upgrade. After upgrading, delete the named connection and add a new named connection. Alternatively, configure the Emulator download service for the existing named connection by manually updating each piece of invalid data for the download service.

Emulator cannot load a .cem file that resides in the platform release directory

To load a .cem file, first copy the .cem file to a different directory and then load the .cem file.

Running the Emulator on a dual-processor machine is slow

If you encounter performance issues while running the Emulator on a development workstation with multiple CPUs, try one of the following approaches to improve performance:

  • In Task Manager, force the Emulator to run on only one CPU by right-clicking the emulator.exe process, choosing Set Affinity, and selecting one CPU.
  • Force Platform Builder, Platform Manager, and the Emulator to run on the same CPU. In Task Manager, right-click the cepb.exe process, choose Set Affinity, and then select a CPU. Repeat for the cemgr.exe and emulator.exe processes, selecting the same CPU.

If these approaches do not increase the performance of the Emulator, try disabling one of your CPUs in BIOS.

Host system must have a NIC installed to use Remote Tools against the Emulator

With the Emulator, to use a remote tool over the TCP/IP transport, a network interface card (NIC) must be installed on the development workstation. If you do not have a NIC, you can install the Microsoft Loopback Adapter instead.

To install the Microsoft Loopback Adapter in Windows XP

  1. Open Control Panel.
  2. Choose Add Hardware.
  3. Choose Next.
  4. Choose Yes, I have already connected the hardware, and then choose Next.
  5. From the Installed hardware list, select Add a new hardware device, and then choose Next.
  6. Choose Install the hardware that I manually select from a list (Advanced), and then choose Next.
  7. From the Common hardware types list, select Network adapters, and then choose Next.
  8. From the Manufacturer list, select Microsoft.
  9. From the Network Adapter list, select Microsoft Loopback Adapter, and then choose Next.
  10. Choose Next.
  11. Choose Finish.

To install the Microsoft Loopback Adapter in Windows 2000

  1. Open Control Panel.
  2. Choose Add/Remove Hardware.
  3. Choose Next.
  4. Choose Add/Troubleshoot a device, and then choose Next.
  5. From the Devices list, select Add a new device, and then choose Next.
  6. Choose No, I want to select the hardware from a list, and then choose Next.
  7. From the Hardware types list, select Network adapters, and then choose Next.
  8. From the Manufacturers list, select Microsoft.
  9. From the Network Adapter list, select Microsoft Loopback Adapter, and then choose Next.
  10. Choose Next.
  11. Choose Finish.

Uninstalling Platform Builder after the Emulator stops responding causes Windows 2000 to stop responding

For a development workstation running Microsoft Windows 2000 Professional, if you want to uninstall Platform Builder, reboot the development workstation before uninstalling the product if you ran the Emulator and the Emulator stopped responding. If you do not reboot the development workstation, you may encounter an error and a system reboot when you attempt to uninstall Platform Builder.

Emulator fails to resolve host names for multihomed hosts

When using the Emulator on a multihomed host machine, networking applications may fail because they send packets out on the wrong network. This problem can be avoided if you disable all the network cards except the one the user wants the Emulator to use, and then reruns the Emulator.

Must customize the Emulator for it to work in an SDK

As a platform developer, you may need to customize an SDK for the Emulator to work. Properties associated with the Universal SDK such as Universal and Parent do not exist in TUSDK_EMULATION. A dialog box displaying a warning may appear when downloading the OS image.

Large OS images can be too big to build or download and can fail

The Japanese OS Image for a Web Pad Enterprise release is too large to build on the Emulator.

As a workaround, do the following:

  1. After creating a new platform, choose Settings from the Platform menu.
  2. Choose the Environment tab.
  3. Create new environment variable IMGRAM64=1.
  4. Build the image.
  5. After the image is built, choose Configure Remote Connection from the Target menu.
  6. In the Download box, choose Emulator, and then choose Configure.
  7. Under System group, set Memory=64 MB.
